1. Призначення та межі системи

Pulso PPG Backend приймає від ESP32 пакети фотоплетизмографічних зразків з двох оптичних каналів: інфрачервоного ir та червоного r. Сервер підтримує живу обробку потоку, короткі вимірювальні сесії, довші записи, трансляцію сирих пакетів для інтерфейсу, збереження записів у PostgreSQL та експорт даних у JSON або CSV.

Основні вихідні фізіологічні оцінки: частота серцевих скорочень bpm, насичення крові киснем spo2, ratio-of-ratios ratio, індекс довіри sensor_confidence і структурований опис якості сигналу signal_quality. Додатково backend повертає waveform_morphology з інженерним описом форми PPG-хвилі, стабільності пульсових циклів і усередненим шаблоном хвилі. Значення можуть бути null, якщо вікно ще коротке, контакт з пальцем не виявлений або сигнал не дає надійної AC/DC оцінки.

Функціональна межа: система оцінює життєві показники з PPG-сигналу на рівні програмного прототипу. Вона не містить клінічного калібрування під конкретну оптичну конструкцію, пацієнтську популяцію, температуру, тон шкіри, рух або медичний стандарт перевірки точності. Морфологічні показники описують форму хвилі та стабільність сигналу, але не діагностують судинний стан або серцево-судинні патології.

2. Архітектура програми

Проєкт складається з двох частин: Python-бекенду та прошивки ESP32. Бекенд приймає WebSocket-повідомлення, валідує JSON, веде ковзні буфери, фільтрує PPG-канали, рахує метрики та керує записами. Прошивка налаштовує MAX30102/MAX3010x, підключається до Wi-Fi, відкриває WebSocket-клієнт і передає батчі зразків.

4. Потік даних від датчика до результату

4.1. Формат вхідного пакета

Сервер очікує JSON, де кореневий об'єкт має поле device. Кожен sample містить інфрачервоний канал ir і червоний канал r. Частота дискретизації передається в полі fs. 

{
  "device": {
    "id": "F4:65:0B:55:2E:80",
    "recording_id": "optional-recording-uuid",
    "temp": 31.75,
    "fs": 25,
    "samples": [
      {"ir": 84231, "r": 53211}
    ]
  }
}

4.2. Валідація

Pydantic-моделі DevicePayload, DeviceData і PPGSample перевіряють, що ідентифікатор пристрою непорожній, fs > 0, фактична частота не нижча за 5 Гц, а зразки каналів додатні. Поле r в JSON мапиться на внутрішнє поле red.

4.3. Ковзне вікно

Для кожного device_id створюється окремий DeviceSignalBuffer. Якщо пристрій змінює fs, буфер очищається, бо зразки з різними частотами не можна коректно змішувати в одному часовому ряді. Максимальна довжина вікна: round(MAX_WINDOW_SECONDS * fs). За стандартних значень це 20 * 25 = 500 samples.

4.4. Результат метрик

Успішно оброблений пакет повертає VitalSigns і оновлює MetricsStore. Під час активного запису backend також додає sample rows у сесію та може транслювати live batch через /recordings/live. Поле waveform_morphology є додатковим і не змінює старі поля bpm, spo2, ratio, sensor_confidence або signal_quality.

6. Збереження записів і модель даних

PostgreSQL вмикається, якщо заданий DATABASE_URL, POSTGRES_DSN або компонентні змінні POSTGRES_HOST, POSTGRES_PORT, POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB. Без БД live-метрики й сесії працюють у пам'яті, але endpoints для recordings/projects/users повертають 503.

Таблиця Ключові поля Призначення

У recordings_samples є унікальне обмеження (recording_id, sample_index), що не дає випадково записати однаковий sample index двічі. Індекси за started_at, status і recording_id прискорюють типові запити.

У фінальному записі signal_quality зберігається як JSON і містить shape_score, якщо morphology була доступна. Повний waveform_morphology з шаблоном хвилі передається в live/API відповіді VitalSigns; окремої колонки для average_pulse_template у таблиці recordings поточна схема не створює.

7. Фізична основа PPG та пульсоксиметрії

7.1. Фотоплетизмографія

Фотоплетизмографія вимірює зміну інтенсивності світла, що проходить через тканину або відбивається від неї. Коли серце викидає кров, артеріальний об'єм у периферичній тканині періодично змінюється; це змінює поглинання та розсіювання світла. PPG-сигнал умовно складається з повільної непульсуючої складової DC і малої пульсуючої складової AC, синхронної з серцевим циклом [1].

У цій системі MAX30102 працює як reflective PPG sensor: червоний та інфрачервоний світлодіоди освітлюють тканину, а фотодетектор вимірює відбите світло. Сенсор також має FIFO, програмовані параметри sample rate, LED current, pulse width, ADC range та внутрішній температурний сенсор [5].

7.2. Чому потрібні два канали

Оксигемоглобін і дезоксигемоглобін мають різні спектри поглинання в червоній та інфрачервоній області. Пульсоксиметрія використовує це співвідношення: зміни AC/DC на двох довжинах хвиль комбінуються у ratio-of-ratios, після чого SpO2 отримують через емпіричну калібрувальну функцію [2], [3].

Фізичний сенс AC/DC:
DC приблизно відповідає сталому оптичному шляху через тканини, венозну кров, непульсуючу артеріальну кров і конструктивні умови датчика.
AC відповідає зміні оптичного сигналу, викликаній серцевим пульсом.
PI = (ACIR / DCIR) * 100%.

7.3. Перфузійний індекс

Perfusion index є відношенням пульсуючої частини сигналу до непульсуючої частини. Він не є самостійним медичним діагнозом, але добре підходить як інженерний індикатор сили PPG-сигналу та якості контакту. У програмі PI використовується для contact detection, дозволу/заборони SpO2 та внеску в sensor_confidence [4].

8. Алгоритми цифрової обробки сигналу

8.1. Hampel suppression для одиничних викидів

Перед фільтрацією кожен канал проходить Hampel-подібну обробку. Для кожного sample у центрі локального вікна розраховується медіана та robust scale через median absolute deviation. Якщо точка відхиляється більше ніж на 3 * MAD, її замінюють медіаною. Це приглушує одиничні spike-и без сильного спотворення форми пульсу [7].

Для вікна Wi:
mi = median(Wi)
MADi = 1.4826 * median(|xj - mi|)
якщо |xi - mi| > 3 * MADi, тоді xi := mi.

8.2. Лінійний detrend

PPG має повільний baseline drift через рух, тиск пальця, дихання, температуру та зміну контакту. scipy.signal.detrend(..., type="linear") віднімає найкращу лінійну складову з поточного вікна, залишаючи швидші коливання, корисні для BPM і AC-оцінки.

xd[n] = xclean[n] - (a n + b), де a і b оцінюються як лінійний тренд на поточному вікні.

8.3. Butterworth band-pass

Після detrend сигнал проходить смуговий фільтр Butterworth 4-го порядку зі стандартною смугою 0.5..3.0 Hz. Верхня частота обмежується 0.95 * Nyquist, щоб не порушити цифрові умови фільтрації. Звичайний режим використовує sosfiltfilt, тобто прямий і зворотний прохід для нульової фазової затримки; якщо вікно замале, код переходить на sosfilt [6], [9].

fNyquist = fs / 2.
Для fs = 25 Hz маємо fNyquist = 12.5 Hz, тому смуга 0.5..3.0 Hz безпечна для цифрової обробки.
3.0 Hz відповідає 180 bpm, бо bpm = 60 * f.

8.4. Пошук піків

Піки шукаються на відфільтрованому IR-каналі. Мінімальна відстань між піками задається через максимальний дозволений пульс: round(fs * 60 / MAX_BPM). При fs=25 і MAX_BPM=180 це приблизно 8 samples. Початкова prominence дорівнює 0.35 * std(filtered_ir). Якщо піків замало, код робить повторний пошук із relaxed threshold на основі MAD.

8.5. Аналіз морфології PPG-хвилі

Аналіз морфології PPG-сигналу розширює функціонал системи, оскільки дозволяє оцінювати не лише BPM та SpO2, а й форму окремих пульсових хвиль. Для цього сигнал розбивається на пульсові цикли, кожен цикл нормалізується за амплітудою та тривалістю, після чого розраховуються часові, амплітудні та кореляційні ознаки. Отримані показники використовуються для додаткової оцінки якості сигналу, виявлення рухових артефактів та передачі усередненої форми хвилі на frontend.

Новий модуль app/processing/morphology.py використовує вже знайдені піки IR-каналу і не дублює фільтрацію або peak detection. Між сусідніми піками шукаються локальні мінімуми, які виконують роль foot points. Один пульсовий цикл визначається як ділянка від мінімуму перед піком до наступного мінімуму після піка. Цикли відкидаються, якщо їхня тривалість виходить за межі MIN_BPM..MAX_BPM, амплітуда надто мала або амплітуда є очевидним outlier-ом.

Для кожного валідного циклу рахуються pulse_amplitude, pulse_duration_ms, rise_time_ms, decay_time_ms, pulse_width_50_ms, rise_slope, decay_slope, area та symmetry_ratio. Числові поля у відповіді є медіанами по валідних пульсах, а не значеннями одного випадкового циклу.

Для порівняння форми кожен цикл ресемплюється до 100 точок і нормалізується за амплітудою так, що локальний мінімум стає 0, а максимум стає 1. З normalized pulses формується average_pulse_template. Далі кожен нормалізований пульс порівнюється з шаблоном через correlation, а медіанне значення кореляції стає shape_similarity. Додатково рахуються amplitude_variability, duration_variability і morphology_variability.

shape_score має діапазон 0..1 і комбінує similarity, стабільність амплітуди, стабільність тривалості, кількість валідних пульсів та варіативність rise time / pulse width. Статус shape_quality може бути stable, moderately_stable, unstable, insufficient_pulses, low_amplitude, irregular_shape або invalid_signal. Ці ознаки описують waveform shape і pulse stability як інженерні характеристики сигналу; вони не є клінічною діагностикою.

8.6. Welch fallback

Якщо часовий пошук піків не дає достатньо валідних інтервалів, система переходить до спектральної оцінки через метод Welch. У діапазоні MIN_BPM/60 .. MAX_BPM/60 вибирається частота з найбільшою потужністю, але тільки якщо пік щонайменше втричі перевищує median noise floor. Додаткова перевірка не дає алгоритму зафіксуватися на другій гармоніці, якщо біля f/2 є порівнянний пік [8], [9].

9. Математична модель BPM, SpO2, морфології і якості сигналу

9.1. Мінімальне вікно

Якщо довжина вікна менша за MIN_WINDOW_SECONDS, система не видає BPM і SpO2. За стандартних значень потрібно мінімум 8 секунд. Confidence у цьому стані масштабується лише до 0.2, а signal_quality.level дорівнює warming_up. waveform_morphology може бути присутнім у відповіді, але при недостатній кількості валідних пульсів shape_score дорівнює null і не впливає на confidence.

window_seconds = N / fs
якщо window_seconds < MIN_WINDOW_SECONDS, тоді confidence = clamp(window_seconds / MIN_WINDOW_SECONDS * 0.2).

9.2. BPM з міжпікових інтервалів

Якщо знайдено достатньо піків, оцінка BPM базується на медіані валідних inter-beat intervals. Інтервали фільтруються за межами пульсу MIN_BPM..MAX_BPM, а потім очищаються від outlier-ів через IQR-правило.

IBIk = (peakk+1 - peakk) / fs
valid IBI: 60 / MAX_BPM <= IBIk <= 60 / MIN_BPM
BPM = 60 / median(valid IBI)

Регулярність ритму оцінюється через coefficient of variation:

CV = std(IBI) / mean(IBI)
regularity_score = clamp(1 - CV / 0.18)

9.3. AC/DC для SpO2

DC-компоненти беруться як середні значення сирих каналів. AC-компоненти оцінюються за медіанною амплітудою між піком і мінімумом наступного сегмента. Якщо піків недостатньо, використовується RMS fallback: 2 * sqrt(2) * RMS(filtered). Для синусоїдального сигналу це наближує peak-to-peak amplitude; у ratio-of-ratios однаковий множник у двох каналах скорочується.

DCIR = mean(raw_ir), DCRed = mean(raw_red)
R = (ACRed / DCRed) / (ACIR / DCIR)
PI = (ACIR / DCIR) * 100

9.4. Калібрувальна формула SpO2

У поточному коді SpO2 рахується квадратичною емпіричною формулою з коефіцієнтами з конфігурації. Це типова інженерна форма для двохвильової pulse oximetry, але її потрібно калібрувати під конкретний датчик, LED current, геометрію корпуса, місце вимірювання та референсний пульсоксиметр [2], [3].

SpO2 = clamp(0, 100, A R2 + B R + C + offset)
Стандартно: A = -45.060, B = 30.354, C = 94.845, offset = -1.8.

9.5. Контакт і якість сигналу

Contact detection захищає від вимірювань без пальця, із засвіченим або насиченим сенсором. Якщо DC нижчий за мінімум, канал насичений або PI за межами очікуваного діапазону, система повертає no_contact, bpm=null, spo2=null і sensor_confidence=0.

Порогове значення Стандарт Сенс
MIN_IR_DC 50000 IR DC занизький: немає достатнього оптичного контакту.
MIN_RED_DC 10000 Red DC занизький.
MAX_SENSOR_DC 250000 Канал імовірно насичений.
MIN_PERFUSION_INDEX 0.05% Мінімум для виявлення контакту.
MIN_SPO2_PERFUSION_INDEX 0.05% Мінімум для публікації SpO2.
MAX_PERFUSION_INDEX 20% Надто великий PI: нестабільний або засвічений сигнал.

9.6. Shape score для морфології

Морфологічний score є окремою евристичною оцінкою стабільності форми хвилі. Він розраховується тільки тоді, коли з IR-каналу можна виділити щонайменше три валідні пульсові цикли. Якщо циклів замало, сигнал невалідний або амплітуда занизька, shape_quality пояснює причину, а shape_score залишається null або дорівнює 0 для непридатної амплітуди.

pulsei,norm = resample((pulsei - min) / (max - min), 100)
template = mean(pulsei,norm)
shape_similarity = median(corr(pulsei,norm, template))
amplitude_variability = std(amplitude) / mean(amplitude)
duration_variability = std(duration) / mean(duration)

Фінальний shape_score комбінує shape_similarity, стабільність амплітуди, стабільність тривалості, кількість валідних пульсів і стабільність додаткових часових ознак. Інтерпретація статусів: 0.8..1.0 — stable, 0.5..0.8 — moderately_stable, 0.2..0.5 — unstable, нижче 0.2 — непридатна або нерегулярна форма. Це службова оцінка waveform shape, а не медична класифікація.

9.7. Sensor confidence

Confidence є евристичним score від 0 до 1. Він не є статистичною ймовірністю. Формула комбінує тривалість вікна, perfusion index, регулярність інтервалів, кількість піків, правдоподібність ratio і, коли доступний, score стабільності форми PPG-хвилі.

base_confidence = 0.20 * window_score + 0.25 * pi_score + 0.30 * regularity_score + 0.15 * peak_score + 0.10 * ratio_score
якщо BPM відсутній: base_confidence := base_confidence * 0.45
якщо SpO2 відсутній: base_confidence := base_confidence * 0.70
якщо shape_score доступний: confidence = 0.85 * base_confidence + 0.15 * shape_score
інакше: confidence = base_confidence

Рівень signal_quality.level стає high, якщо є BPM і SpO2, confidence не нижче 0.7, PI достатній і знайдено щонайменше 6 піків. Інакше система повертає medium або low. Якщо morphology недоступна через коротке вікно або малу кількість пульсів, shape_score залишається null і не штрафує sensor_confidence у стані warming up.

10. Логіка прошивки ESP32

10.1. Налаштування сенсора

Прошивка використовує MAX30105-compatible API для сенсора MAX30102. Поточні параметри: LED brightness 0x1F, sample average 4, режим двох LED 2, hardware sample rate 100 Hz, reported sample rate 25 Hz, pulse width 411 us, ADC range 4096.

Сенсор генерує FIFO-зразки швидше, ніж backend отримує дані. ESP32 вичитує FIFO, але додає у batch лише ті samples, які відповідають програмному інтервалу 1000 / 25 = 40 ms. Це дає backend-у стабільну заявлену частоту 25 Гц.

10.2. WebSocket lifecycle

  1. setup() ініціалізує Serial, MAX30102, Wi-Fi та WebSocket.
  2. Після WStype_CONNECTED пристрій надсилає hello.
  3. Backend реєструє device id і може пізніше надіслати start.
  4. Команда start задає activeRecordingId, тривалість і очищає FIFO.
  5. Під час streaming ESP32 накопичує 25 samples і відправляє batch JSON.
  6. Коли локальний wall-clock duration завершився, ESP32 відправляє останній batch і finished.
  7. Команда stop примусово відправляє поточний batch, повертає stop_ack і чистить стан.

10.3. Пакетування

batchSize=25 означає приблизно один JSON-пакет на секунду при 25 Гц. Це зменшує overhead WebSocket-повідомлень і водночас залишає backend-у достатньо часте оновлення live-графіків. 

{
  "device": {
    "id": "F4:65:0B:55:2E:80",
    "recording_id": "uuid",
    "fs": 25,
    "temp": 25.6,
    "samples": [
      {"ir": 133000, "r": 99300}
    ]
  }
}

10.4. Відновлення після збоїв

Якщо Wi-Fi втрачено, прошивка зупиняє streaming без очищення activeRecordingId і пробує перепідключитися кожні 5 секунд. WebSocket має heartbeat 15 s / 3 s / 2 retries. Якщо WebSocket-відправка не вдалася, streaming також зупиняється, щоб не накопичувати неконсистентний стан.

12. Перевірка, тестування та обмеження

12.1. Автоматичні тести

У tests/test_processing.py створюється синтетичний PPG-сигнал: синусоїда з заданим BPM, DC-рівнями, AC-амплітудами, ratio і шумом. Тести перевіряють основний сценарій, warming-up, no-contact, низький PI для SpO2, завершення measurement session, matching recording id, live payload і наявність waveform_morphology у live metrics. У tests/test_morphology.py окремо перевіряються чисті регулярні пульси, шум із збереженою формою, спотворені пульсові форми, недостатня кількість піків, надто низька амплітуда, різні тривалості циклів і фіксована довжина average_pulse_template, а також безпечна обробка NaN як invalid_signal.

Під час підготовки документа виконано: 

python3 -m unittest discover tests

Результат: 17 tests, OK. Для синтетичного 15-секундного сигналу тестовий прогін отримав приблизно 71.4 BPM, SpO2 96.0%, ratio 0.5547, sensor_confidence 0.970 і shape_score 0.973.

12.2. Інженерні обмеження

  • SpO2 є оцінкою, а не клінічно валідованим значенням; коефіцієнти треба калібрувати на референсних даних.
  • Морфологічні ознаки описують форму PPG-хвилі та стабільність сигналу, але не є діагнозом або клінічною оцінкою стану судин.
  • Алгоритм не має акселерометра або adaptive motion artifact cancellation, тому рух може створювати хибні піки.
  • Morphology analysis залежить від коректного пошуку піків; якщо motion artifact зміщує піки або foot points, форма циклу може бути оцінена неправильно.
  • Двохвильова модель не відрізняє карбоксигемоглобін, метгемоглобін та інші нестандартні оптичні стани крові.
  • Вплив тону шкіри, температури, вазоконстрикції та низької перфузії не компенсується окремою моделлю.
  • WebSocket/REST endpoints у поточному коді не мають автентифікації й авторизації.
  • При перезапуску процесу in-memory MetricsStore і активні сесії втрачаються; записи зберігаються тільки якщо налаштована БД.
  • CSV-експорт розгортає recording-level fields на кожен sample row; це зручно для аналізу, але дублює метадані.
  • Повний average_pulse_template передається у live/API відповіді, але не виділений окремою колонкою у persistence-схемі.

12.3. Рекомендації для подальшої валідації

  • Зібрати парні вимірювання з сертифікованим пульсоксиметром і підігнати SPO2_A/B/C/OFFSET.
  • Додати motion quality score або акселерометричний канал для відкидання рухових артефактів.
  • Перевірити пороги shape_score на реальних записах із рухом, слабким контактом і різною перфузією.
  • Логувати firmware version, LED current, contact pressure proxy і умови вимірювання разом із записом.
  • Додати інтеграційні тести для WebSocket команд start/stop і PostgreSQL persistence.
  • Ввести автентифікацію для API, якщо система використовується поза локальною лабораторною мережею.
